    v0.9.0 Architecture & Security Review — Design Spec

    Date: 2026-03-17 Version: draft-2.0 Target: openclaw-cortex v0.9.0

    Executive Summary

    This spec documents a full architecture and security review of openclaw-cortex v0.8.0 (main branch) and defines the scope for v0.9.0. The review identified 9 security/reliability findings and 5 architectural gaps. v0.9.0 addresses all critical and medium findings, adds Sentry + Prometheus observability, and ships three user-facing features: multi-user namespacing, streaming recall, and batch capture.

    Work is organized into three tracks. Track 1 PRs are fully independent. Track 2 and Track 3 are serialized within each track; across tracks they share overlapping files and must be sequenced to avoid merge conflicts (see PR Sequencing).

    Current Architecture (v0.8.0 Snapshot)

    Stack

    • Language: Go 1.25
    • Storage: Memgraph (unified graph + vector DB, Bolt protocol, neo4j-go-driver)
    • Embeddings: Ollama (nomic-embed-text, 768-dim) or OpenAI (text-embedding-3-small)
    • LLM: Claude Haiku via Anthropic SDK or OpenClaw gateway (OpenAI-compatible endpoint)
    • CLI: cobra + viper
    • Testing: black-box tests in tests/, internal/store/mock_store.go (MockStore), golangci-lint v2

    Runtime Modes

    1. CLI — openclaw-cortex <command> (capture, recall, search, store, lifecycle, etc.)
    2. HTTP API server — openclaw-cortex serve (REST on :8080)
    3. MCP server — openclaw-cortex mcp (stdio transport, tools: remember/recall/forget/search/stats/entity_*)

    Key Internal Packages

    Package Key Types
    internal/memgraph/ MemgraphStore (CRUD + vector search + graph traversal), GraphAdapter (implements graph.Client)
    internal/recall/ Recaller (8-factor scoring, RRF graph merge), Reasoner (LLM re-ranking)
    internal/capture/ ClaudeCapturer, EntityExtractor, ConflictDetector, MemoryContradictionDetector
    internal/graph/ graph.Client interface, FactExtractor, EntityResolver, FactResolver
    internal/api/ HTTP API server with Bearer auth middleware
    internal/mcp/ MCP server (mcp-go, stdio transport)
    internal/hooks/ PreTurnHook, PostTurnHook for Claude Code integration
    internal/llm/ LLMClient interface, AnthropicClient, GatewayClient
    internal/lifecycle/ TTL expiry, session decay
    internal/config/ Viper config with env var overrides and validation
    internal/store/ Store interface, MockStore (tests)

    Note: GraphAdapter lives in internal/memgraph/graph.go, not internal/graph/. The graph.Client interface is defined in internal/graph/client.go.

    Existing Security Controls (preserve)

    • XML-escaping on all LLM prompt inputs in capture, reasoner, conflict detector, fact extractor
    • crypto/subtle.ConstantTimeCompare for Bearer token auth
    • Per-operation context timeouts on all Memgraph sessions (10s read / 30s write)
    • http.MaxBytesReader (1 MB) on all POST handlers
    • Config validation at load time; API key masking in String() methods
    • store.ErrNotFound sentinel for proper error discrimination
    • HTTP server timeouts: ReadHeader 10s, Read 30s, Write 60s, Idle 120s

    Security & Reliability Findings

    Critical

    S1 — GatewayClient has no HTTP timeout - File: internal/llm/gateway.go:23 - NewGatewayClient creates &http.Client{} with no timeout. If the gateway hangs, all LLM-dependent paths (capture, contradiction detection, re-ranking, fact extraction) block indefinitely. - Fix: Add configurable timeout (default 60s) via claude.gateway_timeout_seconds config field. - PR: 1.1

    S2 — HTTP API auth disabled by default, no TLS - File: internal/api/server.go:73, cmd/openclaw-cortex/cmd_serve.go:38 - api.auth_token defaults to "", which disables all auth. The serve command logs a warning but proceeds. There is no TLS support. - Fix: Require explicit --unsafe-no-auth flag to run without auth; add --tls-cert / --tls-key flags. - PR: 1.2

    S3 — Vector dimension hardcoded in DDL - File: internal/memgraph/graph.go:56 - The Memgraph vector index DDL string hardcodes "dimension": 768 independent of memory.vector_dimension config. Schema and config can silently drift. - Fix: GraphAdapter.EnsureSchema accepts a vectorDim int parameter; MemgraphStore.EnsureCollection reads cfg.Memory.VectorDimension and passes it to NewGraphAdapter(...).EnsureSchema(ctx, dim). - PR: 1.3

    Medium

    S4 — Pagination cursor is a raw SKIP integer - File: internal/memgraph/store.go:289, internal/api/server.go:401 - The cursor is a plaintext decimal string encoding the SKIP offset. Clients can forge arbitrary offsets and the implementation detail is exposed. - Fix: Opaque HMAC-SHA256 signed cursor (encodes offset + timestamp). - PR: 1.4

    S5 — No rate limiting on HTTP API - File: internal/api/server.go - No per-IP or global rate limiting. A client can flood embedding generation and Memgraph simultaneously. - Fix: Per-IP token bucket middleware using golang.org/x/time/rate. - PR: 1.2

    S6 — Entity search fetches all results then filters in-process - File: internal/api/server.go:566, internal/store/store.go:52 - handleSearchEntities calls store.SearchEntities(ctx, name) which has no limit parameter, then filters by type and truncates in-process. O(N) for large entity sets. - Fix: Add limit int and entityType string parameters to store.Store.SearchEntities; push filtering into the Cypher query in MemgraphStore. - PR: 1.4

    S7 — Unnecessary re-embed on update when content unchanged - File: internal/api/server.go:322 - handleUpdate re-embeds existing content when no content field is provided in the request. This incurs unnecessary latency and embedding cost. - Fix: Skip re-embed when the content field is empty in the update request. - PR: 1.4

    Low / Documentation

    S8 — SECURITY.md references Qdrant (removed in v0.7.0) - File: SECURITY.md:45 - Out-of-scope section says "Vulnerabilities in Qdrant, Ollama, or the Anthropic API themselves". Qdrant was replaced by Memgraph in v0.7.0. - Fix: Update to reference Memgraph. - PR: 1.4

    S9 — Memgraph URI logged at INFO level - File: internal/memgraph/store.go:59 - logger.Info("connected to Memgraph", "uri", uri) logs the full connection URI which may contain embedded credentials. - Fix: Downgrade to logger.Debug and redact credentials from URI before logging. - PR: 1.4

    Architecture Findings

    A1 — No multi-user namespace isolation - The project string is the only tenant boundary. There is no user_id concept anywhere in models, store, or API. Multiple users sharing one Cortex instance have no isolation. - Required for any shared or team deployment. - PR: 3.1

    A2 — PostTurnHook pipeline is synchronous and monolithic - PostTurnHook runs capture, entity extraction, fact extraction, contradiction detection, reinforcement, and dedup sequentially in one function. Any slow LLM call blocks the whole turn. - Makes latency tuning and failure isolation difficult. - Not in v0.9.0 scope (requires larger refactor). Documented for v1.0.0.

    A3 — No backpressure on LLM calls - Up to 4 serial Claude calls per capture turn. No concurrency control, no circuit breaker. Under load, rate limit errors cascade silently. - Partial mitigation in v0.9.0 via Sentry span visibility; full fix (worker pool + circuit breaker) in v1.0.0. - Not in v0.9.0 scope.

    A4 — Graph traversal depth is global, not per-caller - Depth is set in config and applies equally to hook (latency-sensitive) and CLI (latency-tolerant) callers. - Partial mitigation already in place via separate graph_budget_ms vs graph_budget_cli_ms config fields. No action in v0.9.0.

    A5 — SearchEntities has no limit parameter in Store interface - store.Store.SearchEntities(ctx, name) returns unbounded results. - Fixed as part of S6 (PR 1.4).

    v0.9.0 Scope

    PR 2.2 — feat/prometheus-metrics (~3 files) - Files: internal/metrics/metrics.go (rewrite), internal/api/server.go, cmd/openclaw-cortex/cmd_serve.go - Migration note: internal/metrics/metrics.go currently uses expvar.Int counters exposed at /debug/vars.     Track 3 — Features (3 PRs, serialized)

    PR 3.1 must merge before 3.2 and 3.3 because it changes the Store interface and models.Memory.

    PR 3.1 — feat/user-namespacing (~30 files — split recommended) - Size note: This is the largest PR in the release.     PR 3.3 — feat/batch-capture (~3 files) - Files: cmd/openclaw-cortex/cmd_capture_batch.go (new), internal/capture/batch.go (new), cmd/openclaw-cortex/main.go - Relationship to existing cmd_store_batch.go: store-batch is a low-level bulk-upsert of pre-formed memory records (no LLM extraction). capture-batch is higher-level: it runs the full capture pipeline (LLM extraction, entity extraction, fact extraction) on raw conversation turns. These are different commands; both are kept. - New cortex capture-batch command reading JSONL from stdin - Input format per line: {"user": "...", "assistant": "...", "project": "...", "user_id": "..."} - New capture.BatchCapturer: worker pool, configurable --concurrency (default 4, max 16) - Progress to stderr: processed N/total, stored M memories, skipped K (low quality) - --dry-run: runs extraction, no store writes; prints extracted memories as JSONL to stdout - --stats: prints summary JSON to stdout on completion - Error handling: failed lines logged to stderr with line number; processing continues - Tests: verify concurrency bound; verify dry-run produces no store writes; verify JSONL parse errors non-fatal

    Observability Decision

    Sentry (PR 2.1) handles error tracking and performance tracing — actionable alerts when things break. SaaS, no infrastructure needed. No-op when DSN is unset.

    Prometheus (PR 2.2) handles operational metrics — dashboards, latency trends, capacity planning. Requires a running Prometheus + Grafana stack. The /metrics endpoint is always available regardless.

    These are complementary. Sentry ships first (lower friction, immediate value). The existing expvar metrics in internal/metrics/metrics.go are replaced by Prometheus types in PR 2.2; this is a breaking change to the metrics API but /debug/vars was never documented as stable.

    What is NOT in v0.9.0

    Documented for v1.0.0:

    • A2 — PostTurnHook async refactor: Decompose into a pipeline with per-stage goroutines and a result channel. Requires careful back-pressure design.
    • A3 — LLM circuit breaker: Worker pool + exponential backoff + circuit breaker for all LLM call sites.
    • Web UI memory browser: Deferred; requires frontend build pipeline.
    • Pluggable embedding providers (Cohere, Gemini): Deferred.

    PR Sequencing

    Track 2 and Track 3 share files (internal/api/server.go, cmd_*.go). To avoid merge conflicts, they are serialized:

    Week 1:  1.1, 1.2, 1.3, 1.4  (all parallel — no shared files)
Week 2:  3.1a (model + store layer)
Week 3:  3.1b (API/MCP/CLI) → 2.1 (Sentry)
Week 4:  2.2 (Prometheus) → 3.2 (streaming recall)
Week 5:  3.3 (batch capture) → release v0.9.0

    Track 1 establishes the security baseline and should ship first. PR 3.1a (model layer) unblocks 3.1b and all subsequent PRs that need the UserID field. PR 2.1 (Sentry) touches all cmd_*.go files and must follow 3.1b. PRs 2.2 and 3.2 both modify internal/api/server.go and must be serialized.
