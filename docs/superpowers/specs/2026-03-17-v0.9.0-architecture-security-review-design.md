# v0.9.0 Architecture & Security Review — Design Spec

**Date:** 2026-03-17
**Version:** draft-1.0
**Target:** openclaw-cortex v0.9.0

---

## Executive Summary

This spec documents a full architecture and security review of openclaw-cortex v0.8.0 (main branch) and defines the scope for v0.9.0. The review identified 9 security/reliability findings and 5 architectural gaps. v0.9.0 addresses all critical and medium findings, adds Sentry + Prometheus observability, and ships three user-facing features: multi-user namespacing, streaming recall, and batch capture.

Work is organized into three independent tracks that can be developed and merged as separate PRs on the existing `feat/<topic>` branch model.

---

## Current Architecture (v0.8.0 Snapshot)

### Stack
- **Language:** Go 1.25
- **Storage:** Memgraph (unified graph + vector DB, Bolt protocol, neo4j-go-driver)
- **Embeddings:** Ollama (`nomic-embed-text`, 768-dim) or OpenAI (`text-embedding-3-small`)
- **LLM:** Claude Haiku via Anthropic SDK or OpenClaw gateway (OpenAI-compatible endpoint)
- **CLI:** cobra + viper
- **Testing:** black-box tests in `tests/`, MockStore, golangci-lint v2

### Runtime Modes
1. **CLI** — `openclaw-cortex <command>` (capture, recall, search, store, lifecycle, etc.)
2. **HTTP API server** — `openclaw-cortex serve` (REST on `:8080`)
3. **MCP server** — `openclaw-cortex mcp` (stdio transport, tools: remember/recall/forget/search/stats/entity_*)

### Key Internal Packages
| Package | Responsibility |
|---------|---------------|
| `internal/memgraph/` | MemgraphStore: CRUD + vector search + graph traversal |
| `internal/recall/` | 8-factor scoring, RRF graph merge, Reasoner (LLM re-ranking) |
| `internal/capture/` | ClaudeCapturer, EntityExtractor, ConflictDetector |
| `internal/graph/` | GraphAdapter, FactExtractor, EntityResolver, FactResolver |
| `internal/api/` | HTTP API server with Bearer auth middleware |
| `internal/mcp/` | MCP server (mcp-go) |
| `internal/hooks/` | PreTurnHook, PostTurnHook for Claude Code integration |
| `internal/llm/` | LLMClient interface, AnthropicClient, GatewayClient |
| `internal/lifecycle/` | TTL expiry, session decay |
| `internal/config/` | Viper config with env var overrides and validation |

### Existing Security Controls (preserve)
- XML-escaping on all LLM prompt inputs in capture, reasoner, conflict detector, fact extractor
- `crypto/subtle.ConstantTimeCompare` for Bearer token auth
- Per-operation context timeouts on all Memgraph sessions (10s read / 30s write)
- `http.MaxBytesReader` (1 MB) on all POST handlers
- Config validation at load time; API key masking in `String()` methods
- `store.ErrNotFound` sentinel for proper error discrimination
- HTTP server timeouts: ReadHeader 10s, Read 30s, Write 60s, Idle 120s

## Security & Reliability Findings

### Critical

**S1 — GatewayClient has no HTTP timeout**
- File: `internal/llm/gateway.go:23`
- `NewGatewayClient` creates `&http.Client{}` with no timeout. If the gateway hangs, all LLM-dependent paths (capture, contradiction detection, re-ranking, fact extraction) block indefinitely.
- Fix: Add configurable timeout (default 60s) via `claude.gateway_timeout_seconds` config field.

**S2 — HTTP API auth disabled by default, no TLS**
- File: `internal/api/server.go:73`, `cmd/openclaw-cortex/cmd_serve.go:38`
- `api.auth_token` defaults to `""`, which disables all auth. The serve command logs a warning but proceeds. There is no TLS support.
- Fix: Require explicit `--unsafe-no-auth` flag to run without auth; add `--tls-cert` / `--tls-key` flags.

**S3 — Vector dimension hardcoded in DDL**
- File: `internal/memgraph/graph.go:56`
- The Memgraph vector index DDL string hardcodes `"dimension": 768` independent of `memory.vector_dimension` config. Schema and config can silently drift.
- Fix: Inject dimension from config at `EnsureCollection` call time.

### Medium

**S4 — Pagination cursor is a raw SKIP integer**
- File: `internal/memgraph/store.go:289`, `internal/api/server.go:401`
- The cursor is a plaintext decimal string encoding the SKIP offset. Clients can forge arbitrary offsets and the implementation detail is exposed.
- Fix: Opaque HMAC-SHA256 signed cursor (encodes offset + timestamp).

**S5 — No rate limiting on HTTP API**
- File: `internal/api/server.go`
- No per-IP or global rate limiting. A client can flood embedding generation and Memgraph simultaneously.
- Fix: Per-IP token bucket middleware using `golang.org/x/time/rate`.

**S6 — Entity search fetches all results then filters in-process**
- File: `internal/api/server.go:566`
- `handleSearchEntities` calls `store.SearchEntities` (no limit param), then filters by type and truncates. O(N) for large entity sets.
- Fix: Push type filter and limit into `SearchEntities` / Cypher query.

**S7 — Unnecessary re-embed on update when content unchanged**
- File: `internal/api/server.go:322`
- `handleUpdate` re-embeds existing content when no `content` field is provided in the request. This incurs unnecessary latency and embedding cost.
- Fix: Store embedding alongside the memory node; fetch it on update when content is unchanged.

### Low / Documentation

**S8 — SECURITY.md references Qdrant (removed in v0.7.0)**
- File: `SECURITY.md:45`
- Out-of-scope section says "Vulnerabilities in Qdrant, Ollama, or the Anthropic API themselves". Qdrant was replaced by Memgraph.
- Fix: Update to reference Memgraph.

**S9 — Memgraph URI logged at INFO level**
- File: `internal/memgraph/store.go:59`
- `logger.Info("connected to Memgraph", "uri", uri)` logs the full connection URI which may contain embedded credentials.
- Fix: Downgrade to `logger.Debug` and redact credentials from URI before logging.

## Architecture Findings

**A1 — No multi-user namespace isolation**
- The `project` string is the only tenant boundary. There is no `user_id` concept anywhere in models, store, or API. Multiple users sharing one Cortex instance have no isolation.
- Required for any shared or team deployment.

**A2 — PostTurnHook pipeline is synchronous and monolithic**
- `PostTurnHook` runs capture, entity extraction, fact extraction, contradiction detection, reinforcement, and dedup sequentially in one function. Any slow LLM call blocks the whole turn.
- Makes latency tuning and failure isolation difficult.
- Not in v0.9.0 scope (requires larger refactor), but documented for v1.0.0.

**A3 — No backpressure on LLM calls**
- Up to 4 serial Claude calls per capture turn. No concurrency control, no circuit breaker. Under load, rate limit errors cascade silently.
- Partial mitigation in v0.9.0 via Sentry span visibility; full fix (worker pool + circuit breaker) in v1.0.0.

**A4 — Graph traversal depth is global, not per-caller**
- Depth is set in config and applies equally to hook (latency-sensitive) and CLI (latency-tolerant) callers.
- Minor issue; partial mitigation already in place via separate `graph_budget_ms` vs `graph_budget_cli_ms` config fields.

**A5 — SearchEntities has no limit parameter in Store interface**
- `store.Store.SearchEntities(ctx, name)` returns unbounded results.
- Fix included in S6 above.

## v0.9.0 Scope

### Track 1 — Security & Reliability (4 PRs)

All Track 1 PRs are independent and can be reviewed in parallel.

**PR 1.1 — `fix/gateway-timeout`**
- Files: `internal/llm/gateway.go`, `internal/config/config.go`
- Add `TimeoutSeconds int` to `ClaudeConfig`; default 60
- Pass `http.Client{Timeout: duration}` to `NewGatewayClient`
- Add `claude.gateway_timeout_seconds` / `OPENCLAW_CORTEX_CLAUDE_GATEWAY_TIMEOUT` env binding
- Tests: mock HTTP server that sleeps; verify timeout error is returned

**PR 1.2 — `fix/api-hardening`**
- Files: `cmd/openclaw-cortex/cmd_serve.go`, `internal/api/server.go`
- Serve command: add `--unsafe-no-auth` flag; fail-fast if auth token empty and flag not set
- Serve command: add `--tls-cert` / `--tls-key` flags; call `httpSrv.ListenAndServeTLS` when both provided
- New `internal/api/ratelimit.go`: per-IP token bucket middleware (`golang.org/x/time/rate`); configurable via `api.rate_limit_rps` (default 100) and `api.rate_limit_burst` (default 20)
- Tests: verify 429 on burst, verify auth rejection, verify TLS handshake with self-signed cert

**PR 1.3 — `fix/schema-dim-from-config`**
- Files: `internal/memgraph/graph.go`, `internal/memgraph/store.go`
- Pass `vectorDim int` parameter to `EnsureSchema`; use `fmt.Sprintf` to inject into DDL string
- `MemgraphStore.EnsureCollection` reads `cfg.Memory.VectorDimension` and passes it down
- Tests: assert DDL contains correct dimension string

**PR 1.4 — `fix/misc-hardening`**
- Files: `internal/memgraph/store.go`, `internal/api/server.go`, `SECURITY.md`, `pkg/cursor/cursor.go` (new)
- New `pkg/cursor/cursor.go`: HMAC-SHA256 signed cursor encoding `{skip, timestamp}`; `Sign(skip int64, secret []byte) string` and `Verify(cursor string, secret []byte) (int64, error)`
- `api.cursor_secret` config field (auto-generated on first run if empty, stored in config)
- `handleList` uses `cursor.Sign` / `cursor.Verify`; returns 400 on tampered cursor
- `memgraph/store.go:59` downgrade to `logger.Debug`, redact credentials from URI
- `internal/api/server.go:322` skip re-embed when content field is empty in update request
- `SECURITY.md` update Qdrant → Memgraph reference
- Tests: verify tampered cursor returns 400; verify same-secret round-trip

### Track 3 — Features (3 PRs)

**PR 3.1 — `feat/user-namespacing`** *(largest PR)*
- Files: `internal/models/memory.go`, `internal/store/store.go`, `internal/memgraph/store.go`, `internal/store/mock_store.go`, `internal/api/server.go`, `internal/mcp/server.go`, `internal/hooks/hooks.go`, `internal/config/config.go`, all `cmd_*.go` files
- Add `UserID string` field to `models.Memory` and `models.Entity`
- Add `UserID *string` to `store.SearchFilters`
- Memgraph schema: `CREATE INDEX ON :Memory(user_id)`, `CREATE INDEX ON :Entity(user_id)`
- All Cypher WHERE clauses include `user_id` filter when set
- Config: `user.default_user_id` (default `"default"`); `--user` flag on all commands
- HTTP API: read `X-User-ID` header (or query param `user_id`); falls back to config default
- MCP: `user_id` optional parameter on `remember`, `recall`, `search`, `forget` tools
- Hooks: read `session_id` as user scope when no explicit user set
- Migration: `cortex migrate --add-user-id` backfills `user_id = "default"` on existing nodes
- Tests: verify cross-user isolation (user A cannot recall user B's memories)

**PR 3.2 — `feat/streaming-recall`**
- Files: `internal/api/server.go`, `internal/recall/recall.go`
- New `GET /v1/recall/stream` endpoint; accepts same query params as `POST /v1/recall`
- Response: `Content-Type: text/event-stream`; each SSE event is one ranked memory JSON object
- Flush after each memory so client can start rendering immediately
- Graceful fallback: if `Accept` header is `application/json`, behaves identically to existing `POST /v1/recall`
- Recaller emits results via channel; handler reads and flushes
- Tests: verify SSE event format; verify ordering matches non-streaming recall

**PR 3.3 — `feat/batch-capture`**
- Files: `cmd/openclaw-cortex/cmd_capture_batch.go` (new), `internal/capture/batch.go` (new)
- New `cortex capture-batch` command reading JSONL from stdin
- Input format: `{"user": "...", "assistant": "...", "project": "...", "user_id": "..."}` per line
- New `capture.BatchCapturer`: worker pool with configurable `--concurrency` (default 4, max 16)
- Progress output to stderr: `processed N/total, stored M memories, skipped K (low quality)`
- `--dry-run`: runs extraction but does not write to store; prints extracted memories as JSONL
- `--stats`: prints summary JSON to stdout on completion
- Error handling: failed lines logged to stderr with line number; processing continues
- Tests: verify concurrency bound; verify dry-run produces no store writes; verify JSONL parse errors are non-fatal

## Observability Decision

**Sentry** (PR 2.1) handles error tracking and performance tracing — actionable alerts when things break. It is a SaaS service requiring a DSN; when DSN is unset it is a no-op with zero overhead.

**Prometheus** (PR 2.2) handles operational metrics — dashboards, latency trends, capacity planning. It requires a running Prometheus + Grafana stack to be useful. The `/metrics` endpoint is always available; whether it is scraped is the operator's choice.

These are complementary. Sentry ships first because it has lower operational friction (no infrastructure required) and provides immediate value for error investigation.

---

## What is NOT in v0.9.0

These items are documented for v1.0.0:

- **A2 — PostTurnHook async refactor**: Decompose into a pipeline with per-stage goroutines and a result channel. Requires careful back-pressure design.
- **A3 — LLM circuit breaker**: Worker pool + exponential backoff + circuit breaker for all LLM call sites.
- **Web UI memory browser**: Deferred; requires frontend build pipeline.
- **Pluggable embedding providers** (Cohere, Gemini): Deferred.

## PR Sequencing

```
Week 1:  1.1, 1.2, 1.3, 1.4  (parallel — all independent)
Week 2:  2.1 (Sentry) + 3.1 (user namespacing, largest)
Week 3:  2.2 (Prometheus) + 3.2 (streaming recall)
Week 4:  3.3 (batch capture) → release v0.9.0
```

Track 1 PRs unblock nothing but should ship first to establish the security baseline. Track 3.1 is the largest change (touches every package) and should be reviewed with extra care. Tracks 2 and 3 are independent of each other and can be reviewed in parallel.
